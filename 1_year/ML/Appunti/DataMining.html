The Data Mining Process

Business Intelligence

Process of:

  1. Transforming raw data into useful information to support effective and aware business strategies
  1. Capturing the business data and getting the right information to the right people, at the right time, through the right channel

Business Intelligence and Data Warehouse

Data Warehouse is an optimized repository that stores information useful for decision-making by providing the ability to manage sets of historical data which are fundamental for activities like forecasting as an example.

It :

  • is subject-oriented
    focuses on enterprise-specific concepts
  • is integrated and consistent
    integrates data from different and heterogeneous sources and provides a unified view of all the data
  • is not volatile and shows evolution over time
    the changes in the data are tracked and recorded showing them over time.

OLTP and OLAP

  • OLTP
    1. For transaction
    1. Each transaction reads a few records
    1. Frozen workload
  • OLAP
    1. For dynamic multidimensional analyses
    1. Each query involves huge amount of records
    1. Mutable workload

ETL (Extraction Transformation and Loading)

We extracts, integrates, and cleans data from operational sources to feed the Data Warehouse layer.

ETL phases

  • Extraction

    We have different data types (structured or not) and different extraction types:

    • static → snapshot of data
    • incremental → regular update → timestamp and triggers
  • Cleansing

    Quality improvement procedure, standardize:

    • Duplicate data
    • Missing data
    • Improper field use
    • Wrong values
    • Inconsistent values for error or bad usage
  • Transformation
    • Conversion → data type change
    • Enrichment → attribute combination
    • Separation/Concatenation → attribute denormalized/normalized
  • Loading
    • Refresh → rewrite of DWH (ok with static extraction)
    • Update → rewrite only changed data (ok with progressing extraction)

DWH Architecture

We require:

  • Separation → analytical ≠ transactional
  • Scalability → system upgrades with data volume
  • Extensibility → can host new applications/technologies
  • Security → monitoring accesses
  • Administrability → easy to manage

We have:

  • Single-Layer

    ✅ Limited space
    ❌ No sepratation

  • Two-Layer

    ✅ Separation

  • Three -Layer



DFM (Dimensional Fact Model)

It is a conceptual model created specifically to function as data mart design support. It is graphic and based on the multidimensional model.

The basic concepts are:

ConceptDescription
FactIt is a concept relevant to decision-making processes that models a set of events.
MeasureIt is a numerical property of a fact, describes quantitive aspects.
DimensionsIt is a fact property with a finite domain and it’s used for analysis coordination
Dimensional attributeAttributes of dimensions
HierarchyThree of Dimensional attribute that models a many-to-one relation between pairs of dimensional attributes.
There is a dimension in the root and the nodes describe it.
Primary eventParticular occurrence of a fact and provides various dimensions, each with n-ple of values.
Secondary eventSums up all the values of the same measure in the corresponding primary events

Logical Design

Defines the data structure that represent data marts according to the preselected logical model and optimizes performance by fine- tuning these structures.

There are 3 steps to implement a logical schema:

  1. Translating: fact schema → logic schema
    • Star schema
    • Snowflake schema
  1. Materializing views: secondary view that aggregate primary view data to improve query performance
  1. Fragmenting fact tables vertically and horizontally

Data Lake

A Data Lake is a repository of data stored in a raw form.

It will provide data that will be used when they are necessary.

It is also cheaply scalable and quickly ingest everything.

Data could come from batches but also from streams (Internet) so a data lake must also handle queues to stay efficient.

It doesn’t force schema on write, so it doesn’t impose a schema to the data but uses the data own schema.

It can be enabled in less time than a data wharehouse

time to value: from when we need something to when i’ll have a software for that ‘something’

Time to value must be the shortest possible.

Mission critical → complete their mission as soon as possible

Modern Data Architecture

Data Lake Users

They are different and they have different needs:

  • Business Users → pre-configured dashboards
  • Business Analiysts → own model to provide business insights
  • Data Scientists, Engeneers → statistical analysis, implementations, optimizations

Data Lake Components

Orchestration, makes the software components cooperate correctly.

There are 3 phases:

  • Data Ingestion

    Ingestion has to be guaranteed independently from the type of data flow it faces.

    It can come as a workload, as an incremental ingestion or a streaming.

    Change Data Capture (CDC) may be useful when we are receiving many data logs and application events in a continuous ingestion of the data lake.

    To insert data we have code-tools and no-code-tools. The first ones are more flexible but more difficult, the oppositre for the others.

    No-code-tools have smaller time to value but also less value, the opposite for code-tools. It’s a trade-off.
  • Data Storage

    We have 3 zones of storing:

    1. Raw → immutable, self-descriptive with metadata, ok for disaster recovery
    1. Optimized → optimizes format for speed of query
    1. Analytics → clean data for ML
  • Data Processing and Analytics

    The data got extracted and then analized in real-time (Real time depends on the time scale of my system)

Metadata in Data Lake

Helps users to understand and use the data efficiently.

  • They got many components
    • data Source → info on where it was originated
    • data Schema → stucture of data
    • data Format → encoding of files
    • data Quality Metrics → % of missing values or statistic accuracy
    • data Lifecycle → how much time in the data lake
    • data Ownership → who has to manage/maintain the data
    • data Lineage → tracks data lineage
    • Access Control → who has access to the data
    • data Classification → sensitivity or classification level of the data
    • data Usage Information → how many times / who accesses the data

The CRISP-DM methodology

Business understanding

We reformulate the problem until my problem formulation and scenario fit. I have to consider the current situation and where i want to go.

Data unserstanding

I have to understand which data are available and which ones fit the purpose and how much do they cost.

In order to do that in the best way, i need to collect the data, describe and explore them an then i need to verify the quality.

Data preparation

We manipulate the data, transforming it (converting formats and data types) in irder to imporve the quality of the result.

In order to do that we take a data set, select data in it, clean the new dataset, construct the data an then we integrate and format it.

Modeling

We choose a modeling technique and a test design, then build the model and assess it.

Evaluation

We have to make a rigorous assessment of the results of the data mining process and compare different choices on a qualitative and quantitative basis.

Daployment

We make a deployment plan, review the project and deploy it.
